‚óŹ I now have the full crypto surface mapped. Here's my evidence-based answer to
  your go-qrllib question, then I'll kick off a thorough audit.

  Verdict: yes for the signing core, as a gated phase 2 ‚ÄĒ not a drop-in

  Why it makes sense:
  - go-qrllib is the canonical consensus crypto. I confirmed the real chain node
  (go-qrl, vendored here) signs/verifies through
  github.com/theQRL/go-qrllib/crypto/ml_dsa_87 + .../wallet/ml_dsa_87. Your
  current @theqrl/wallet.js + @theqrl/mldsa87 are JS re-implementations of that
  exact library. Compiling go-qrllib to WASM means the wallet runs the same 
  bytes the chain validates, eliminating the "is the JS port faithful?" risk
  entirely. That's the strongest possible argument for it.
  - Go‚ÜíWASM genuinely satisfies "crypto outside TS/JS": the signing math runs as
  WASM; TS only marshals bytes.

  The caveats that make it phase 2, not now:
  1. Bundle weight. Go's stdlib target (GOOS=js GOARCH=wasm) ships the Go
  runtime+GC (multi-MB). TinyGo gets to ~100s of KB but has stdlib/reflection
  limits ‚ÄĒ go-qrllib's SHAKE/keccak + array math is likely TinyGo-clean, but
  that must be verified by building, not assumed.
  2. Async init. WASM instantiation is async, so today's synchronous
  signWithScheme() / mnemonic helpers become async (partly already are, via the
  worker).
  3. ML-KEM-768 gap. go-qrllib is a signatures+hashing library; the relay KEM
  (@noble/post-quantum ml-kem768 in PQCrypto.ts) is almost certainly not in it.
  So go-qrllib gets you the signing+hashing core out of JS, but the relay KEM
  stays JS unless separately addressed. "All crypto outside JS" is only fully
  reachable for the consensus core this way.
  4. Parity is non-negotiable. A swap needs KAT vectors + cross-check against
  @theqrl/wallet.js output + at least one chain-accepted signature, before it
  ships.

  So the plan is two phases, and phase 1 is needed regardless of the engine 
  decision ‚ÄĒ consolidating all crypto behind one audited facade is exactly what
  makes the engine swappable later, and it's where the TS hardening + launder
  removal live.

  Let me run a comprehensive parallel audit to produce the exact worklist + a
  grounded go-qrllib feasibility report before I touch code.

  Neither specifically ‚ÄĒ and that's the key insight. The 64-byte address is a
  unified, algorithm-agnostic format:

  address = SHAKE256( descriptor ‚ÄĖ publicKey )[:64]

  The 3-byte descriptor (first byte = wallet type: 0x01 ML-DSA-87, 0x00
  SPHINCS+-256s/reserved) is hashed into the address, so the address
  cryptographically commits to which signature scheme it belongs to ‚ÄĒ but the

  bytes (512-bit) restores ~256-bit, matching ML-DSA-87's NIST Level 5. It
